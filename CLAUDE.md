# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A two-step pipeline that turns the Oslo municipality property spreadsheet
(`Oversikt-over-Oslo-kommunes-eiendommer-mai-2026_nett-2.xlsx`, 7,022 rows) into
an interactive Leaflet map of municipally-owned land/buildings, split by owning
agency (Eiendoms- og byfornyelsesetaten, Oslobygg KF, Boligbygg Oslo KF, Oslo
Havn KF). See `README.md` for the data-source caveat: this is *only* the
municipality's own slice of the cadastre — it cannot answer "what's privately
owned" without pulling the full Matrikkel register.

```
XLSX ──(geocode.py)──> eiendommer.geojson ──(index.html)──> map in browser
```

## Commands

Dependencies are Poetry-managed, `package-mode = false` (script-only, not a
distributable package).

```bash
poetry install
poetry run python geocode.py "Oversikt-over-Oslo-kommunes-eiendommer-mai-2026_nett-2.xlsx"
# → eiendommer.geojson, geocode_cache.sqlite, missing.csv

poetry run python -m http.server 8000   # serves index.html
```

There are no tests, no linter config, and no build step.

## Architecture

### `geocode.py` — XLSX → GeoJSON

The matrikkelnummer in the `Eiendom` column (`KNR-GNR/BNR/FNR/SNR`, KNR always
`0301`) is the join key. The script:

1. Parses every row's matrikkel via `MATRIKKEL_RE`.
2. **Deduplicates to ~6,636 unique `(gnr, bnr)` pairs** before hitting the
   network — important, because Geonorge is a free public API and the row count
   would otherwise be ~7,022.
3. Looks each pair up via Geonorge's adresse-søk
   (`ws.geonorge.no/adresser/v1/sok`) — no API key, returns WGS84 directly via
   `utkoordsys=4326`. Multiple matching addresses are averaged on their
   `representasjonspunkt` so big multi-address parcels land near the centre.
4. **Caches every lookup in `geocode_cache.sqlite`** keyed on `(gnr, bnr)`.
   Re-runs are instant and resumable; deleting the cache forces a re-fetch.
   Misses are cached too (as `source = "none"`), so they don't keep retrying.
5. Joins coordinates back to every row → writes `eiendommer.geojson` (Points)
   and `missing.csv` (rows with no resolvable address).

Retry/backoff lives in `geonorge_lookup` (4 attempts, 1.5s × attempt for
429/5xx). `SLEEP = 0.05s` between calls — be polite if you tune this.

### The road-land gap

~2,200 rows geocode cleanly; the other ~4,800 are mostly **veigrunn** (road
land) and unregistered land with no street address. They land in `missing.csv`.
To map those, you need **parcel geometry**, not an address — Kartverket's
*"Matrikkelen – Eiendomskart Teig"* WFS, keyed on the matrikkelnummer. The
geocoder is structured so a second resolver can be slotted in without touching
the rest: cache table `geo` already has a `source` column; add new rows with a
different `source` value.

### `index.html` — the map

**Generated by `geocode.py`** from the `INDEX_HTML` template constant near the
top of the script — regenerated on every run, so edits made directly to
`index.html` are overwritten. If you want to change the map UI, edit
`INDEX_HTML` in `geocode.py`, not the HTML file. (No build step; the file is
self-contained.) Leaflet + markercluster + leaflet.heat from CDN, CARTO dark
basemap, three views (Punkter / Tetthet / Areal-vekt), per-agency toggles,
bydel filter, live stats. Falls back to four embedded sample features when
`eiendommer.geojson` isn't fetchable, so the page is useful before geocoding
finishes.

The pipeline contract: `eiendommer.geojson` is a FeatureCollection of Points
with properties `eiendom`, `adresse`, `bydel`, `eier`, `bruksnavn`, `areal`.
The `OWNERS` constant in the HTML is the closed set of recognised agency names
(also drives the colour palette) — keep it in sync with the values that appear
in the `Eiers/festers kontaktinstans` column.

## Conventions

- The XLSX column names in `cols` (`geocode.py:121-128`) are the contract with
  the source spreadsheet. If a new spreadsheet drops with renamed columns, this
  is the single place to update.
- `geocode_cache.sqlite` is committed-adjacent (gitignored or not, depending);
  treat it as durable data, not throwaway. A full re-geocode of 6.6k pairs at
  50ms each is ~5–6 minutes plus Geonorge's own latency.
- Output GeoJSON uses `[lon, lat]` order (GeoJSON spec) even though the cache
  stores `(lat, lon)` — see the swap at `geocode.py:173`.
